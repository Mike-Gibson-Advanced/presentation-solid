Overview

  • They aim to make it more likely for a developer to create a system that is easy to maintain and extend over time

“ S ” Single Responsibility Principle

A class should have only a single responsibility, and that responsibility should be entirely encapsulated by the class

“ O ” Open/Closed Principle

“Software entities (classes, modules, functions, etc.) should be open for extension, but closed for modification”
Bertrand Meyer

Open/Closed Principle

  1. Open for extension This means that the behavior can be extended. That we can make it behave in new and different ways as the requirements of the application change, or to meet the needs of new applications.
  2. Closed for modification The source code cannot be changed.

“ L ” Liskov Substitution Principle

If S is a subtype of T, then objects of type T may be replaced with objects of type S without altering any of the desirable properties of that program.
Basically, it should be possible to use a subtype of a class, and there should be no unexpected behaviour.

Liskov Substitution Principle

Imposes the following:

  • Contravariance of method arguments in the subtype
  • Covariance of return types in the subtype
  • Subtype shouldn't throw any new exceptions except where those exceptions are themselves subtypes of exceptions thrown by the supertype
  • Preconditions cannot be strengthened in a subtype.
  • Postconditions cannot be weakened in a subtype
  • Invariants of the supertype must be preserved in a subtype
